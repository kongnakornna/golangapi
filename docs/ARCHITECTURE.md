# การออกแบบสถาปัตยกรรม (Architecture Design)
## เป้าหมายและหลักการ
- Domain-First (เน้นโดเมนเป็นหลัก): รวมศูนย์ความสามารถทางธุรกิจ (Business Logic) ไว้ที่ internal/core เพื่อรักษาความเกาะกลุ่มของโค้ดให้สูง (High Cohesion)
- Application Assembly Isolation (แยกส่วนการประกอบแอปพลิเคชัน): แต่ละแอปพลิเคชันจะมี Composition Root แยกเป็นอิสระต่อกันที่ internal/apps/<app> โดยรองรับการใช้โมดูลร่วมกันระหว่างหลายแอปพลิเคชัน
-  Replaceable Infrastructure (โครงสร้างพื้นฐานที่ปรับเปลี่ยนได้): รวบรวมส่วนประกอบด้านเทคนิค เช่น ฐานข้อมูล (Database), แคช (Cache) และคิว (Queue) ไว้ที่ internal/platform เพื่อให้ง่ายต่อการเปลี่ยนผ่านหรือแก้ไข
- Unified Transport Layer (เลเยอร์การรับส่งข้อมูลที่เป็นหนึ่งเดียว): ใช้เครื่องมือสำหรับจัดการ Request/Response และ Middleware ร่วมกันเป็นมาตรฐานเดียวที่ internal/transport

## เป้าหมายและหลักการ
- **เน้นโดเมนเป็นหลัก:** ความสามารถทางธุรกิจถูกรวมไว้ใน `internal/core` เพื่อรักษาความสอดคล้องในระดับสูง
- **การแยกส่วนประกอบของแอปพลิเคชัน:** แต่ละแอปพลิเคชันมีรากการประกอบที่เป็นอิสระ (`internal/apps/<app>`) รองรับการใช้งานโมดูลร่วมกันของหลายแอปพลิเคชัน
- **การเปลี่ยนโครงสร้างพื้นฐานได้:** ฐานข้อมูล/แคช/คิว ฯลฯ ถูกรวมศูนย์ไว้ใน `internal/platform`
- **เลเยอร์การขนส่งแบบรวมศูนย์:** เครื่องมือและมิดเดิลแวร์สำหรับการร้องขอ/ตอบสนองแบบรวมศูนย์ (`internal/transport`)

## ภาพรวมโครงสร้างไดเรกทอรี
```text
internal/
  apps/
    app/
      bootstrap/      # การประกอบแอปพลิเคชัน: DI, การเริ่มต้น, จัดการวงจรชีวิต
      router/         # การประกอบเส้นทางของแอปพลิเคชัน
  core/
    user/             # ตัวอย่างโมดูลโดเมน
      handler/
      service/
      repository/
      dto/
      model/
      routes.go
    auth/
    health/
  platform/
    config/           # การโหลดและตรวจสอบการตั้งค่า
    db/               # การเชื่อมต่อฐานข้อมูลและ Redis
  transport/
    httpx/            # การครอบคลุม request/response
    middleware/       # มิดเดิลแวร์
```
## หน้าที่ที่แบ่งเป็นชั้นๆ

- **bootstrap (root path)**: สร้างการพึ่งพา เริ่มบริการ และจัดการวงจรชีวิต
- **router (การประกอบเส้นทาง)**: ประกอบเส้นทางโมดูล สายโซ่มิดเดิลแวร์ และเส้นทางเวอร์ชัน
- **core (โมดูลโดเมน)**: ตรรกะทางธุรกิจและแบบจำลอง เน้นที่อินเทอร์เฟซอย่างเคร่งครัด
- **platform (โครงสร้างพื้นฐาน)**: การพึ่งพาภายนอกและการกำหนดค่า ใช้โดยชั้นการประกอบแอปพลิเคชัน
- **transport (ชั้นการขนส่ง)**: ข้อกำหนดการร้องขอ/ตอบสนอง HTTP แบบรวม และความสามารถในการทำงานข้ามแง่มุม
- 
## การเชื่อมต่อ การส่งข้อมูล ตัวกลาง

```
cmd/app/main.go

-> internal/apps/app/bootstrap/app.go
-> internal/apps/app/router
-> internal/transport/middleware
-> internal/core/<domain>/handler
-> internal/core/<domain>/service
-> internal/core/<domain>/repository
-> internal/platform/db

```

## ทิศทางการพึ่งพา

- `core` ไม่พึ่งพา `apps` หรือ `platform`

- `apps` พึ่งพา `core`, `platform` และ `transport`

- `transport` สามารถพึ่งพา `pkg` ได้ แต่ไม่สามารถพึ่งพา `core` ได้ ## การบันทึกและการติดตาม

- ใช้ `pkg/logger` อย่างสม่ำเสมอและส่งคืน `trace_id` ในการตอบกลับ


- ใช้รูปแบบเดียวกันสำหรับบันทึกคำขอและข้อผิดพลาดเพื่อหลีกเลี่ยงการใช้ระบบบันทึกหลายระบบ

## มาตรฐานการตอบสนอง
โครงสร้างการตอบสนองที่เป็นมาตรฐาน:
```json
{ "code": 200, "message": "OK", "data": {...}, "trace_id": "..." }
```
ในกรณีข้อผิดพลาด ฟิลด์ `data` จะประกอบด้วยประเภทข้อผิดพลาดและข้อมูลฟิลด์
 
## Redis และกลยุทธ์แคช
- Redis ควบคุมผ่านสวิตช์ `app.redis.enabled`

เมื่อ Redis ใช้งานไม่ได้ จะเป็น Noop โดยอัตโนมัติ（แคชและคิวจะลบออกอย่างเงียบ ๆ）


## Redis และกลยุทธ์การทำ Cache
- การควบคุม: Redis ถูกควบคุมผ่านสวิตช์การตั้งค่า app.redis.enabled
- การจัดการเมื่อระบบขัดข้อง: เมื่อ Redis ไม่สามารถใช้งานได้ ระบบจะทำการ Downgrade (ลดระดับ) เป็น Noop โดยอัตโนมัติ (ซึ่งจะส่งผลให้ Cache และ Queue หยุดทำงานอย่างเงียบๆโดยไม่แจ้งข้อผิดพลาด)

## การขยายแอปพลิเคชันหลายตัว
โครงสร้างที่แนะนำเมื่อเพิ่มแอปพลิเคชันใหม่:
```text
cmd/app2/main.go
internal/apps/app2/bootstrap
internal/apps/app2/router
``` 
มดูลโดเมนยังคงใช้ `internal/core` ร่วมกัน
